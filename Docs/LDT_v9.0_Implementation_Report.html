Section 1
EXECUTIVE SUMMARY

LDT v9.0.0 introduces the Enterprise Governance Edition -- a policy-driven execution framework that transforms the diagnostic toolkit from a standalone analysis tool into a governed, auditable, fleet-ready platform. Two new PowerShell modules -- GovernanceEngine.psm1 and FleetGovernance.psm1 -- add structured policy profiles, transaction control with crash recovery, five distinct execution modes, exception governance with configurable abort thresholds, and directory-level tamper detection with whitelist enforcement.

  • Enterprise Governance Engine with 3 policy profiles (Strict, Balanced, Aggressive) and full transaction control
  • Fleet Governance with directory tamper detection via VersionManifest.json and remediation whitelist enforcement
  • 5 execution modes -- AuditOnly, Diagnostic, Remediation, Full, ClassifyOnly -- for granular operational control
  • 7th GuardEngine gate added for whitelist enforcement before any remediation action executes
  • Crash recovery via ExecutionState.json with automatic session restoration (24-hour max age)
  • 12 compliance artifacts (up from 10) -- adds GovernanceReport.json and ExceptionLog.json
  • Option 57: Governance Audit -- standalone governance health check and policy compliance report
Section 2
NEW MODULES
GOVERNANCEENGINE.PSM1  NEW  ~565 lines  |  15 exported functions  |  6 groups
Group 1 -- Policy Engine
POLICY PROFILES
Initialize-PolicyEngine
Get-PolicyProfile
Test-PolicyGate
Three pre-defined governance profiles control execution behavior across the entire pipeline. Strict: max escalation L1, whitelist required for all remediation. Balanced: max escalation L1, retry enabled with exponential backoff. Aggressive: max escalation L2, allows component-level fixes.
Group 2 -- Transaction Control
CRASH RECOVERY
Save-ExecutionState
Restore-ExecutionState
Clear-ExecutionState
ExecutionState.json checkpoint written at every phase boundary. If a session is interrupted (power loss, crash, forced close), the next run auto-detects the state file, validates age (24-hour max), and offers to resume from the last completed phase. State includes session GUID, completed phases, accumulated findings, and partial results.
Group 3 -- Execution Modes
5-MODE GATING
Get-ExecutionMode
Test-ModeAllowsPhase
Test-ModeAllowsRemediation
Test-ModeAllowsClassification
Each mode defines which phases execute and what actions are permitted. AuditOnly: Phases 0-5,8 (scan + report, no classification or remediation). Diagnostic: Phases 0-5,8 (scan + classify). Remediation: Phases 0,6-8 (fix pre-classified issues). Full: Phases 0-8 (everything). ClassifyOnly: Phases 0-5,8 (classify without fixing).
Group 4 -- Exception Governance
SEVERITY ROUTING
Write-GovernedException
Get-ExceptionSummary
Test-ExceptionThreshold
All exceptions are routed through severity classification -- Warning, Error, or Critical. Configurable abort thresholds (default: 3 critical, 10 errors) trigger automatic session termination with a full exception log export. Prevents runaway failures from cascading through the pipeline.
Group 5 -- Retry Governance
EXPONENTIAL BACKOFF
Invoke-GovernedRetry
Wraps any remediation action with configurable retry logic. Exponential backoff with jitter (1s / 2s / 4s base delays) prevents thundering herd on fleet deployments. Configurable max retries and backoff multiplier per policy profile. Only active in Balanced and Aggressive profiles.
Group 6 -- Business Impact
IMPACT SCORING
Get-BusinessImpactWeight
Calculates business impact weight using a three-factor formula: Category multiplier (Hardware=3, Boot=2.5, Security=2, Performance=1.5) times Severity multiplier (Critical=3, High=2, Medium=1.5, Low=1) times Classification multiplier (L3=3, L2=2, L1=1). Used for prioritization in fleet-wide reporting and KPI aggregation.
FLEETGOVERNANCE.PSM1  NEW  ~436 lines  |  6 exported functions
Directory Integrity
TAMPER DETECTION
Test-DirectoryIntegrity
Validates the entire LDT file system against VersionManifest.json. Detects missing files, unexpected files, and hash mismatches. Reports integrity status (Clean / Tampered / Degraded) with per-file detail. Blocks execution in Strict profile if tampering is detected.
Whitelist Enforcement
APPROVED ACTIONS
Get-RemediationWhitelist
Test-WhitelistApproval
Loads the approved remediation action list from config.json. Every Phase 6 dispatch is validated against the whitelist before execution. Unapproved actions are blocked and logged as governance violations. The 7th GuardEngine gate enforces this check at the engine level.
Rollback Validation
SIMULATION
Test-RollbackSimulation
Validates rollback token BeforeState data without executing any rollback. Confirms that registry exports, restore points, and driver snapshots exist and are readable. Reports rollback readiness as READY / PARTIAL / FAILED.
Fleet KPIs
AGGREGATION
Export-FleetKPIs
Aggregates fleet-wide performance metrics from TrendStore: L1 auto-fix success rate, enterprise score delta over time, issue recurrence rate, and mean time to resolution. Exports FleetKPIs.json for dashboard consumption.
Governance Summary
HEALTH SNAPSHOT
New-FleetGovernanceSummary
Consolidated governance health snapshot combining directory integrity, whitelist compliance, exception summary, policy profile status, and KPI highlights into a single structured report. Used by Option 57 (Governance Audit) for standalone governance health checks.

Section 3
SMART DIAGNOSIS ENGINE INTEGRATION

The 9-phase Smart Diagnosis Engine gains governance awareness at every boundary. Phase 0 initializes the governance subsystem -- loading the active policy profile, setting the execution mode, checking for crash recovery state, and validating directory integrity. Every phase transition writes an ExecutionState.json checkpoint. Phase 6 remediation is gated by the 5-mode system. Phase 7 stress validation monitors exception thresholds. Phase 8 exports two new compliance artifacts: GovernanceReport.json and ExceptionLog.json.

5 EXECUTION MODES
AUDITONLY
Phases 0-5, 8
Scan and report only. No classification, no remediation.
DIAGNOSTIC
Phases 0-5, 8
Scan and classify findings. No remediation.
REMEDIATION
Phases 0, 6-8
Fix pre-classified issues. Requires prior diagnostic run.
FULL
Phases 0-8
Complete pipeline. Scan, classify, fix, validate, report.
CLASSIFYONLY
Phases 0-5, 8
Classify findings without executing any fixes.

PHASE 0 -- GOVERNANCE INIT
PREFLIGHT + GOVERNANCE

  • Get-ExecutionMode (determine phase gates)
  • Restore-ExecutionState (crash recovery check)
  • Test-DirectoryIntegrity (tamper detection)
  • All existing Phase 0 preflight checks
  • Save-ExecutionState checkpoint
PHASES 1-5 -- DATA COLLECTION
CHECKPOINT AT EVERY BOUNDARY
  • ExecutionState.json written after each phase

  • Auto-abort if thresholds exceeded
  • All existing scan logic unchanged

  • 7th GuardEngine gate (whitelist check)

CONFIGURATION CHANGES

Four new sections added to config.ini, bringing the total from 29 to 33 sections. The config.json governance block adds structured policy definitions, execution mode mappings, remediation whitelist, retry policy parameters, and exception thresholds.

config.ini -- 4 New Sections
INI ADDITIONS
  • [GovernanceEngine] -- Profile, mode defaults, state file path
  • [ExceptionGovernance] -- Max critical, max errors, log path
  • [RetryPolicy] -- Max retries, base delay, backoff multiplier, jitter
  • [FleetGovernance] -- Manifest path, whitelist enforcement toggle, KPI export
config.json -- Governance Block
JSON ADDITIONS
  • policies: Strict, Balanced, Aggressive definitions
  • executionModes: Phase maps for all 5 modes
  • whitelist: Approved remediation actions list
  • retryPolicy: Backoff parameters per profile
  • exceptionThresholds: Abort limits per severity
