Core Objective -- Understand This First

Laptop Diagnostic Toolkit v9.0.0 is a deterministic, state-driven diagnostic and remediation engine for Windows 10 and Windows 11 laptops. Runs 100% from a USB drive — no installation, no internet, no cloud, no telemetry. The Smart Diagnosis Engine (Option 54) runs a 9-phase analysis with decision tree logic, creates rollback backups before any fix, validates fixes under stress, and produces structured JSON + HTML reports. The Classification Engine (Option 56) adds formal 3-level L1/L2/L3 diagnostic triage with a 5-branch decision tree. The Governance Audit (Option 57) provides enterprise policy enforcement with Strict/Balanced/Aggressive profiles and transaction control. Includes 45 core diagnostic modules, 9 enterprise engine modules, fleet scanning, and guided quick-start workflows.

Core Architecture
THE 9-LAYER TOOLKIT ARCHITECTURE
0
Layer
LAUNCHER -- Laptop_Master_Diagnostic.bat
Single-click entry point. Checks admin privileges, auto-elevates via UAC, validates PowerShell version (5.1+), sets USB drive path. Presents a 58-option menu (Options 0-57) organized by category. Dispatches to the appropriate PS1 script based on selection.
1
Layer
CORE ENGINE -- Laptop_Diagnostic_Suite.ps1 (7,002 lines)
The foundation. Contains 45 diagnostic and repair modules across 8 categories: System Info, Hardware Tests, Boot/OS Repair, Security, Network, Performance, Refurbishment, and Enterprise checks. Options 1-48. Every module writes structured audit logs and creates restore points before destructive operations.
2
Layer
SMART DIAGNOSIS ENGINE -- Smart_Diagnosis_Engine.ps1 (2,816 lines)
Option 54. The 9-phase orchestrated root cause analysis engine with decision tree logic. Collects a full system snapshot, classifies hardware vs software faults, ranks root causes by weight, auto-remediates L1 software issues, validates fixes under stress, and produces JSON + HTML reports. This is the core of what this guide documents.
3
Layer
QUICK START -- Quick_Start.ps1 (1,382 lines)
Options 51-53. Three guided workflows: Auto-Discover (runs key diagnostics automatically), Quick Fix by Symptom (user picks the problem, tool fixes it), and Score This Machine (weighted 0-100 health score with letter grade A-F). Baseline comparison with JSON snapshots per device serial.
4
Layer
TEAM SCANNER -- Team_Issue_Detector.ps1 (849 lines)
Option 49. Fleet-wide scanning for the three most common team issues: BSOD patterns (minidump + BugCheck analysis), unexpected resets (Kernel-Power 41 + Event 6008), and VPN configuration problems. Generates CSV for fleet aggregation.
5
Layer
FLEET AGGREGATOR -- Fleet_Aggregator.ps1 (394 lines)
Option 50. Merges and deduplicates CSV results from multiple machines into a single fleet view. Generates a self-contained HTML dashboard with sortable table, score distribution chart, and severity breakdown. Identifies fleet-wide patterns.
5b
Layer
ENTERPRISE ENGINES -- Core\ (9 modules)
Reusable .psm1 modules: GuardEngine (7-gate remediation authority), IntegrityEngine (SHA256 tamper detection), ScoringEngine (weighted 0-100 health score), TrendEngine (90-session historical tracking), ComplianceExport (ISO/SOC2/CIS artifacts), ClassificationEngine (3-level L1/L2/L3 diagnostic triage with 5-branch decision tree), GovernanceEngine (enterprise governance with policy profiles, transaction control, crash recovery, 5 execution modes), FleetGovernance (fleet-wide governance with directory tamper detection, whitelist enforcement, rollback simulation, KPI export), and LDT-EngineAdapter (bridge module). Loaded at Phase 0 with graceful degradation if absent.

All thresholds, paths, and behavior switches in one INI file. 25 configurable sections including: General, Diagnostics, HardwareTests, Repair, Fleet, Security, SMART thresholds, Refurbishment, AutoRemediation, SmartDiagnosis, Display, ClassificationEngine, QuickStart scoring weights, and more. No code changes needed to tune behavior.

This step-by-step field guide, team guide with visual walkthrough, version history in CHANGELOG.md, and project context. All documentation ships on the USB alongside the scripts. Fonts are bundled for offline HTML rendering.
State Machine
02
Step-by-Step Execution
THE 9-PHASE STATE MACHINE
Phase
0
PREFLIGHT -- Establish Safe Baseline Before Anything Runs
Every pre-condition is verified before a single diagnostic executes. If any preflight check fails, execution aborts cleanly -- no partial state is left on the machine.
  • Verify Admin -- confirms elevated privileges via WindowsPrincipal IsInRole check; aborts if not elevated
  • Verify PowerShell version -- requires 5.1 minimum; blocks and reports on older versions
  • Generate session GUID -- unique identifier per run, stored in diagnostic state for traceability
  • Create System Restore Point -- "LDT SmartDiag Pre-Execution" checkpoint (if SR service running)
  • Export registry -- HKLM\SYSTEM and HKLM\SOFTWARE exported to Backups\SmartDiag_[timestamp]_preflight\
  • Backup BCD -- bcdedit /export to preflight backup folder
  • Write manifest -- session GUID, timestamp, machine name, PS version, backup contents
Rollback Protection Abort on failure Non-destructive
Phase
1
SNAPSHOT -- Complete System State Collection
Collects a full system snapshot using CIM/WMI queries. All data is stored in the diagnostic state and later serialised to snapshot.json on the USB. Subsequent phases reference this snapshot for baseline comparison.
  • Machine identity -- computer name, serial number, manufacturer, model
  • OS -- version, build number, activation status (Licensed/Unlicensed), pending update count
  • BIOS/UEFI -- BIOS version, firmware type (UEFI vs Legacy via PEFirmwareType registry)
  • Chassis -- chassis type (Laptop/Desktop/Tablet/Convertible) via Win32_SystemEnclosure
  • Hardware -- total RAM (GB), physical disk count, battery presence
  • Uptime -- days/hours/minutes since last boot, power source (AC/Battery)
  • Event log summary -- Error events from System and Application logs (configurable days)
  • Driver inventory -- all PnP signed drivers with name, version, manufacturer, signed status, INF
  • Service inventory -- all services with name, display name, status, start type
Read-Only CIM/WMI Queries Output: snapshot.json
Phase
2
HARDWARE INTEGRITY CLASSIFICATION -- The Hardware vs Software Gate
The most critical phase. If hardware faults meeting the critical threshold are detected, the engine blocks all software remediation (Phases 3-7 skip) and escalates directly. This prevents masking hardware failures with software fixes.
  • SMART disk health -- Get-PhysicalDisk health status + Get-StorageReliabilityCounter for uncorrectable errors and wear %
  • SSD/NVMe wear -- wear level threshold: >80% = Fail, sets HardwareCritical flag
  • WHEA memory errors -- System event log scan for WHEA events (configurable days); 5+ events = Fail
  • Thermal sensors -- MSAcpi_ThermalZoneTemperature; >85C = ThermalCritical, >75C = Warning (configurable)
  • Battery wear -- design vs full charged capacity; >60% wear = Fail, >40% = Warning
  • TPM status -- presence and readiness check via Get-Tpm
  • PnP device errors -- devices with ConfigManagerErrorCode != 0; 3+ = Fail, 1-2 = Warning
# Phase 2 hardware gate -- if critical hardware detected, software phases are skipped
if ($diagState.HardwareCritical) { # Phases 3-7 check this flag and skip }
# Phase 6 (Remediation) only executes if hardware is not critical
Gates Phases 3-7 Weighted Scoring Hardware priority over Software
Phase
3
OS & BOOT INTEGRITY -- Corruption Detection
Validates the integrity of the Windows boot chain and operating system. All checks are read-only -- results feed Phase 6 fix decisions. Skips entirely if Phase 2 flagged HardwareCritical.
  • BSOD analysis -- scans Minidump folder + BugCheck events (Event ID 1001); extracts faulting driver names and stop codes
  • Unexpected resets -- Kernel-Power Event ID 41 (dirty shutdown) + Event ID 6008 (unexpected shutdown)
  • SFC integrity -- reads CBS.log tail for "Cannot repair" or "Corruption not repaired" patterns
  • DISM CheckHealth -- detects component store corruption; flags if RestoreHealth will be required
  • Secure Boot status -- enabled/disabled/not supported (Legacy BIOS)
  • Windows Update age -- days since last installed hotfix; warns if >60 days
  • BCD validation -- bcdedit /enum all; flags missing Windows Boot Manager or Boot Loader entries
  • CHKDSK dirty bit -- fsutil dirty query C:; flags if volume needs CHKDSK on next reboot
  • Corrupt services -- auto-start services with ImagePath pointing to non-existent binaries
  • WU component health -- validates SoftwareDistribution folder + WU service chain (wuauserv, BITS, CryptSvc, msiserver)
Skips if HW Critical Read-Only Feeds Phase 6 fix decisions
Phase
4
SERVICE & DRIVER INTEGRITY -- Fault Attribution
Identifies which specific drivers, services, and security configurations are responsible for instability. Skips if HardwareCritical.
  • Problem devices -- Win32_PnPEntity with ConfigManagerErrorCode != 0; lists device names and error codes
  • Unsigned drivers -- Win32_PnPSignedDriver where IsSigned = false; counts and lists
  • Critical services -- checks WinDefend, mpssvc (Firewall), wuauserv (WU), BITS are running
  • Defender status -- real-time protection enabled, signature age (warns if >7 days)
  • Firewall profiles -- all 3 profiles (Domain, Private, Public) checked independently
  • Network adapters -- active adapter count, Wi-Fi vs Ethernet breakdown
  • Suspicious scheduled tasks -- non-Microsoft tasks with paths in Temp, AppData, Downloads, or cmd.exe /c patterns
  • Display adapter health -- GPU enumeration (Win32_VideoController), ConfigManagerErrorCode, driver age vs threshold
  • GPU TDR/crash events -- Event 4101/4116 (display driver timeout recovery), LiveKernelEvent (GPU hangs); 10+ = Fail
  • Display panel indicators -- Graphics Kernel errors, Video Scheduler Event 129, resolution anomaly detection, internal vs external monitor count
Skips if HW Critical Security + Display surface Read-Only
Phase
5
PERFORMANCE PROFILING -- Baseline & Network Stack
Measures system performance and stores baselines for Phase 7 stress comparison. Also checks network stack health. Skips if HardwareCritical.
  • CPU benchmark -- 10-second iteration count (math.sqrt loop); stored as PerfBaseline.CPUScore
  • Memory pressure -- % used; Fail >90%, Warning >80%
  • Disk free space -- C: drive; Fail <10%, Warning <20%
  • Disk I/O benchmark -- write + read 50MB test file; MB/s stored for Phase 7 comparison
  • Startup items -- Win32_StartupCommand + registry Run keys; Warning if >15 items
  • Default gateway -- Get-NetRoute + ping test; warns if unreachable or missing
  • Winsock catalogue -- netsh winsock show catalog; Fail if corrupt or empty
  • Proxy configuration -- registry check; warns if proxy enabled with no server configured
Skips if HW Critical Stores PerfBaseline Read-Only
Phase
6
CONTROLLED REMEDIATION -- Risk-Ordered Execution
The only phase that modifies the system. Three gates must pass before any fix executes: hardware must not be critical, escalation level must be within MaxAutoLevel (default L1), and confidence must meet the threshold (default 60%).
# Pre-remediation backup (restore point + SERVICES.reg + SOFTWARE.reg + driver CSV + BCD)
New-SmartDiagBackup -BackupDir $BackupPath -DiagState $diagState

# Dispatches to core engine modules by finding category:
BootRepair (SFC/DISM) - BSODTroubleshooter - DriverRepair
SecurityHardening - NetworkTroubleshooter - SoftwareCleanup
Before ANY change: a full backup is created -- System Restore point, registry hives (SERVICES + SOFTWARE), driver version CSV, and BCD export. Each fix result is recorded as SUCCESS or FAILED in the diagnostic state.
Only phase that modifies Rollback before every fix Confidence-gated L1 software fixes only
Phase
7
STRESS VALIDATION -- Post-Fix Stability Confirmation
After remediation, the engine stress-validates the system under load. A PASS means the machine is genuinely stable -- not just that the fix commands ran without error. Skips if HW/Thermal Critical.
  • Pre-stress thermal baseline -- reads MSAcpi_ThermalZoneTemperature before any stress begins
  • CPU stress -- configurable duration (default 60s) math loop; score compared to Phase 5 baseline with tolerance %
  • Memory allocation test -- allocates configurable MB (default 200MB); spot-checks data integrity at 10 random positions
  • Disk I/O stress -- write + read test file; MB/s compared to Phase 5 baseline; flags if degraded beyond tolerance
  • Post-stress event log -- checks for new System errors since Phase 6 timestamp; >5 = Fail
  • Memory Diagnostic recommendation -- if WHEA errors were detected in Phase 2, recommends mdsched.exe on next reboot
  • Post-stress thermal -- reads temperature again, calculates delta; warns if delta >20C or temp exceeds throttle threshold
Baseline Comparison PASS / FAIL verdict Instability = immediate escalation
Phase
8
FINAL CLASSIFICATION -- Root Cause Determination & Report Generation
The engine produces the final deterministic classification using weighted root cause ranking, then generates all output files. Always runs, even when hardware is critical.
  • Primary root cause -- highest-weighted fault that explains the pattern of dependent issues across all phases
  • Secondary symptoms -- 2-4 additional detected faults, ordered by weight descending
  • Enterprise score -- 0-100 composite health score (100 = perfect; deducts per finding weight)
  • Confidence % -- derived from primary fault weight relative to total findings weight (capped 95%)
  • Escalation level -- None / L1 Auto-Fixable / L2 Firmware / L3 Hardware (see Section 04)
  • Terminal summary -- color-coded phase results grid with enterprise score progress bar
  • Outputs written -- snapshot.json + final_report.json + HTML summary report (auto-opens in browser)
Always Runs Output: final_report.json Output: HTML summary
Risk Model
03
Weighted Risk Model
SEVERITY SCORING MODEL -- 0 to 100

Every finding is assigned a weight (1-95). The enterprise score starts at 100 and deducts per finding (Fail = weight x 0.5, Warning = weight x 0.2). Priority: Hardware > Boot/OS > Drivers > Security > Performance.

95
SMART Critical / WHEA Memory
Uncorrectable disk errors, drive status = Unhealthy, or WHEA memory events. Hardware failure -- blocks all software remediation. Sets HardwareCritical flag. L2/L3 escalation.
80
BSOD / Crash Pattern
Multiple BSODs detected (minidumps + BugCheck events). Faulting driver identified from crash data. >=5 crashes = critical threshold. Phase 6 dispatches BSODTroubleshooter.
70
BCD Missing / Boot Corruption
bcdedit /enum shows missing Windows Boot Manager or Boot Loader entry. Boot chain integrity compromised. Phase 6 dispatches BootRepair.
65
GPU TDR / Display Crash Events
10+ display driver timeout recovery events or GPU hangs. May cause repeated restarts and display artifacts. If no physical damage: check eDP cable, then panel replacement (L2).
60
Display Adapter Error
GPU reports ConfigManagerErrorCode != 0. Hardware-level display failure. L2 escalation -- driver repair or hardware replacement.
55
Security Configuration Failure
Defender real-time protection disabled, firewall profiles down, or critical services stopped. Auto-fixable -- Phase 6 dispatches SecurityHardening.
40
DISM Component Store Corrupt
DISM /CheckHealth reports corruption requiring RestoreHealth. Auto-fixable -- Phase 6 dispatches BootRepair (DISM + SFC).
35
Winsock / Network Corrupt
Winsock catalogue corrupt, default gateway unreachable, or proxy misconfigured. Auto-fixable -- Phase 6 dispatches NetworkTroubleshooter.
25
Corrupt Services / Dirty Volume
Auto-start services with missing binaries, or CHKDSK dirty bit set. Warning-level findings that indicate OS maintenance needed.
15
Minor Configuration Drift
Startup overload (>15 items), outdated Defender signatures (>7 days), WU services disabled, or old Windows Update. Auto-fixable in Phase 6.
Escalation Taxonomy
04
3-Level Escalation Model
ESCALATION TAXONOMY
Level 1
AUTO-FIXABLE (SOFTWARE)
  • Defender real-time protection re-enable
  • Firewall profiles re-enable (Domain + Private + Public)
  • Network stack reset (Winsock + IP + adapter cycle)
  • DISM RestoreHealth + SFC repair
  • Windows Update component reset (SoftwareDistribution + catroot2)
  • Driver repair via core engine DriverRepair dispatch
  • BSOD troubleshooting via BSODTroubleshooter dispatch
  • Startup cleanup via SoftwareCleanup dispatch
  • Security hardening via SecurityHardening dispatch
Level 2
REPLACEABLE COMPONENT
  • Battery wear >40% (critical >60%) — replace battery pack
  • SMART critical / drive Unhealthy — replace SSD or HDD
  • NVMe wear >80% (critical >90%) — drive replacement required
  • SSD wear >70% — drive replacement recommended
  • WHEA correctable RAM errors — reseat or replace DIMM
  • CMOS battery dead (clock drift >120s) — replace CR2032 coin cell
  • Display dead pixels >3 — panel replacement per pixel policy
  • Display artifacts (no physical damage) — reseat eDP cable or replace LCD panel
  • Power-on hours >5 years — proactive replacement recommended
Level 3
TECHNICIAN REQUIRED
  • WHEA uncorrectable memory errors — board-level diagnosis
  • No POST — dead system board replacement
  • BIOS corruption — external programmer or Crisis Recovery
  • TPM absent on TPM 2.0-required machine
  • Storage controller failure — NAND unreadable
  • SMART failure — immediate drive replacement
  • L2 escalation triggers: uncorrectable memory, storage controller fail, firmware corruption, TPM hardware fail

Acceptance Criteria
07
Enterprise Acceptance Targets
PERFORMANCE TARGETS — v9.0.0 ACCEPTANCE CRITERIA
≥70%
Software Auto-Resolution Rate
At least 70% of all software and configuration issues auto-resolved by Phase 6 without technician intervention. Issues outside L1 scope are clearly separated as L2 or L3 escalations with specific remediation guidance.
≥90%
Hardware / Software Classification Accuracy
Phase 2 hardware gate must correctly separate hardware-root-cause faults from software-fixable ones with ≥90% accuracy. False positives that block valid software remediation are measured separately and must be below 5%.
≥98%
Rollback Success Rate
Every action taken by Phase 6 remediation must be reversible. Pre-remediation backups (restore point + SERVICES.reg + SOFTWARE.reg + driver CSV + BCD) ensure at least 98% of applied changes can be fully reverted to pre-fix state.
<5%
False-Positive Auto-Fix Rate
Fewer than 5% of auto-applied fixes may target issues that did not genuinely exist. Confidence-gated remediation (default 60% threshold) and weighted scoring prevent single-event false triggers. All findings tracked in the audit log.
